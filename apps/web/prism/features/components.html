01 / 05 · Componentization

Components that build themselves.

The defining law of Prism, and the reason everything else works: if an element has a child, it is a component. You build; the structure appears. No declaring, no naming ceremony, no library housekeeping.

The problem

The component ceremony

In canvas tools, a component is bureaucracy. You draw a button, select it just right, invoke create component, name it, file it into a library — and then maintain that library forever.

Skip the ceremony and your design silently rots into drawings that merely look like a UI: no structure, nothing reusable, nothing editable on its own. The tool made organization your job, and the job never ends.

The approach

Structure is the tool's job

Prism is built on the atomic-design idea — atoms grow into molecules grow into organisms — but it refuses to make you do the assembling twice. You already express structure the moment you put one element inside another. That's all the information a component needs.

The tree of your design is a live, honest reflection of what you actually built.
Componentization at work
Law · if it has a child, it is a component

One rule, applied everywhere

Every element on a Prism frame is a real HTML/CSS element. The moment one contains another, the parent becomes a component: it can be opened alone, edited in isolation, versioned, and shared as a linked instance. A lone text node or an empty box stays a simple element — no noise in your tree.

  • The Wrapper — a rectangle stripped of every default, existing purely to wrap. The neutral container primitive.
  • Mock-first media — images and icons drop in as believable placeholders, so structure reads instantly before assets exist. Or drop a real file straight onto the frame.
  • Scope by birthplace — a component belongs to the workspace, project, screen, or component where it was born. One owner, always.

The result

Always organized — because it was never your job.

You never stop to tidy, never rebuild a library, never wonder which rectangle is the real button. The hierarchy previews, the child lists, the navigation — all fed by the same tree you created just by designing.
